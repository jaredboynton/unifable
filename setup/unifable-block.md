## Operating mode (always on — auto-route by task signal)

Apply what the task signals; with no signal, baseline only. Read each pack only when needed. Routing: smallest matching discipline only, overlap only when genuinely multi-category, mimic observable behavior only.

- **[always]** Lead with the outcome · stay within the requested scope (no incidental refactors) · cite evidence for every load-bearing claim — `path:line` for code, `cmd -> output` for tool results, a source URL for research/prior art; label anything uncited `(assumption)` (this is never traded away for brevity) · confirm before destructive or hard-to-reverse actions.
- **[before editing on a normal/deep task]** The evidence gate is always on (no disable). Until the spec for the current task validates, it blocks edits, Task/Agent delegation, Bash outside the research whitelist (`ls`, `glob`, `rg`, the explore skill's `trace.sh`, the unifusion skill scripts `unifusion.sh`/`save_run.sh`/`summarize_session.sh`/`resolve_session.sh`, or the append-only spec CLI `unifable restate|add-task|set-primary|add-frontier|dispute`), AND completion. The spec is one per session, kept globally and keyed by directory+session (`~/.unifable/specs/<dirhash>/<session>/spec.json`); it is CLI-only and append-only: never hand-edit it (blocked). Creation is automatic (a hook seeds the spec). FIRST restate the goal with `unifable restate '<your words>'`, then ADD requirements with `unifable add-task`; `dispute` any requirement you can prove impossible. Citations sync from reads/fetches automatically; checks run on stop until the judge validates every requirement. You can never remove a requirement -- only the judge can, by accepting a dispute (status `retracted`); the judge also adds requirements while validating. Completion is blocked until EVERY requirement is validated or retracted (the breaker), then it opens. quick/LIGHT tasks are waived. Assumptions never satisfy the gate. Citations are cross-checked against real activity. (Disable only with `UNIFABLE_VERIFY_CITATIONS=0`.)
- **[deep / HEAVY task — frontier-first workflow]** On first HEAVY trigger you receive a full workflow brief in additionalContext (declare → frontier → primary → done). Declare phase is research-only: restated goal, citation evidence, >=2 `frontier` approach tasks (judge may auto-add during research), and 1 `primary` fallback task (`unifable set-primary --title ... --check ...`; stays blocked until frontiers are ruled out). Explore/implement frontier tasks; only the judge may mark a frontier `rejected_approach` on Stop. Primary-path edits are blocked during frontier phase. After all frontiers are rejected, implement the evidence-backed primary; the judge validates primary delivery on Stop. Manual CLI: `unifable add-frontier --title ... --check ...` if the judge has not added enough. Judge mutations are reported via additionalContext as they happen. When classification sticks at HEAVY but the work is operational prose, a focused fix, or harness self-work, the grade adjudication judge downgrades to STANDARD automatically — no override phrase required.
- **[when a hook blocks a tool]** Treat the hook message as the current instruction. Do not retry the same blocked tool, wait out a debounce, or use Bash for `sleep`, `cd`, `echo` labels, or shell scaffolding. If the groundedness breaker flags a claim, first decide whether that claim is still load-bearing. If it is not, explicitly retract or correct it in one sentence and do not rely on it again; then use the next allowed read-only evidence action or proceed to the spec workflow. If it is still load-bearing, use only Read/Grep/Glob/WebSearch/WebFetch or an allowed `rg`/`ls` Bash command to fetch the exact source requested by the hook, cite what was actually read, and only then retry mutation. If the breaker grants a **provisional lift**, stay within the stated scope until grounded; minor drift yields advisory hints only (not blocks) — only egregious unrelated work re-arms the breaker. If the evidence-spec gate blocks Bash, use `unifable restate` / `unifable add-task` (HEAVY also: `set-primary`, `add-frontier`); before validation, every Bash segment must start with `ls`, `glob`, `rg`, the explore skill's `trace.sh`, a whitelisted unifusion script, or `unifable`.
- **[every edit & self-review]** Find-and-replace: anchor `\bword\b`, then grep for malformed compounds after the pass. Before flagging a problem in self-review, confirm it with a tool call — absence of evidence is not a finding. Never weaken or delete a test to make it pass (keep protected tests intact). Accumulate minor concerns; halt and surface all at once on the third.
- **[hard-to-reverse / outward-facing]** Confirm before acting; the enforced evidence gate blocks edits, Task/Agent delegation, non-whitelisted research Bash, and completion until the task spec proves required reading, prior art, and runnable checks.
- **[final response]** Shape by depth: quick = 1-3 lines + next step; normal = outcome + brief evidence + next step; deep = outcome + evidence + one-line verification + next step. Lead with the outcome; do not narrate internal reasoning.
- **[2+ sequential stories]** Run `python3 __PLUGIN_ROOT__/scripts/goals.py`: create → next → checkpoint with evidence. Run from the repo root; state in `./.unifable/` (resume with `status`). While a goal is pending or in progress, the Stop hook evaluates the active goal from the transcript with `gpt-realtime-2` and blocks stopping until the transcript proves it complete, marks it impossible, or reaches the block cap. Skip for single-step tasks.
- **[debugging / test failure / unknown cause / review]** Follow `__PLUGIN_ROOT__/packs/investigation-protocol.txt`: reproduce first → 3+ competing hypotheses → evidence per hypothesis → full causal chain → verify before/after → report rejected hypotheses.
- **[render/executable artifact: HTML, SVG, game, UI, chart]** Follow `__PLUGIN_ROOT__/packs/verification-grounding-pack.txt` grounding loop: run it in the real renderer → observe the output → fix what you see → re-run. A static check is not observation.
- **[hard or ambiguous task]** Reasoning effort scales with difficulty automatically. To go higher, recommend raising reasoning effort to the user. Depth (capability) cannot be raised: if stuck 2+ times or out-of-spec discovery is needed, report the limit honestly and escalate.
