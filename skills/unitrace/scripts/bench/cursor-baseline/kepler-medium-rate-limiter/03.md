Tracing the gateway rate limiter from route handling through headers and fallback behavior.
The gateway rate limiter is a **Cloudflare Durable Object (DO) token bucket** wired into the **X-API-Key auth path only**. JWT Bearer requests bypass it entirely. Here is the full path from route handling through headers and fallbacks.

## Where it sits in request handling

The Worker entry is `gateway/src/index.ts`. After public/unauthenticated routes are handled, authenticated traffic splits:

1. **Bearer JWT** (lines 772–835): validate JWT + scopes, proxy to backend — **no rate limiting**.
2. **X-API-Key** (lines 838+): validate key + scopes, then **rate limit → cache → backend proxy**.

```838:901:gateway/src/index.ts

          rlHeaders = r.headers;
          limited = !r.allow;
          if (rlMode === 'enforce' && !r.allow) return new Response('Too Many Requests', { status: 429, headers: errorCorsHeaders() });
        }
      } catch (e) {
        // Rate limiter timeout or error - log but allow request to proceed (fail open)
        console.warn('Rate limiter error, allowing request', { ... });
        // Continue without rate limiting headers
      }
    }
```

## Configuration (`wrangler.toml` + `limitsFromEnv`)

The DO is bound as `RATE_LIMITER` in `gateway/wrangler.toml`. Limits and mode come from env vars:

| Env var | Default | Purpose |
|---------|---------|---------|
| `RL_MODE` | `observe` | `off` \| `observe` \| `enforce` |
| `RL_DEFAULT_LIMIT` | `1000` | Requests/min per API key (non-SSE) |
| `RL_CHAT_LIMIT` | `100` | Chat stream starts/min per API key |
| `SSE_CONCURRENCY_LIMIT` | `5` | Max concurrent `/v1/chat/stream` per key |

```543:551:gateway/src/index.ts
function limitsFromEnv(env: Env) {
  return {
    cacheEnabled: (env.CACHE_ENABLED || 'false').toLowerCase() === 'true',
    rlMode: (env.RL_MODE || 'off').toLowerCase() as 'off' | 'observe' | 'enforce',
    sseCap: Number(env.SSE_CONCURRENCY_LIMIT || '5'),
    defaultPerMin: Number(env.RL_DEFAULT_LIMIT || '1000'),
    chatPerMin: Number(env.RL_CHAT_LIMIT || '100'),
  };
}
```

**Mode behavior:**

- **`off`**: skip the DO entirely.
- **`observe`**: call the DO, record `limited` for analytics, but **never return 429** — request proceeds even when `allow: false`.
- **`enforce`**: return **429** immediately when `allow: false`.

## Per-client isolation (DO routing)

Each API key maps to one DO instance via SHA-256 hash:

```337:340:gateway/src/index.ts
async function sha256hex(s: string): Promise<string> {
  const buf = await crypto.subtle.digest('SHA-256', te.encode(s));
  return [...new Uint8Array(buf)].map(b => b.toString(16).padStart(2, '0')).join('');
}
```

```555:567:gateway/src/index.ts
async function rlConsume(env: Env, apiKeyHash: string, defaultPerMin: number) {
  const id = env.RATE_LIMITER.idFromName(apiKeyHash);
  const stub = env.RATE_LIMITER.get(id);
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), 2000);
  try {
    const res = await stub.fetch('https://do/check', {
      method: 'POST',
      body: JSON.stringify({ type: 'consume', limits: { defaultPerMin } }),
      signal: controller.signal
    });
```

The Worker posts JSON to the DO stub; the DO's `fetch` handler dispatches on `type`: `consume`, `open`, or `close`.

## Durable Object algorithm (`gateway/src/rate_limiter.ts`)

The `RateLimiter` class holds two independent token buckets plus a stream set, all persisted in DO storage:

```24:34:gateway/src/rate_limiter.ts
export class RateLimiter {
  state: DurableObjectState;
  storage: DurableObjectStorage;

  // Token bucket and stream state, hydrated from Durable Object storage on first request and persisted after each operation
  tokensDefault = 0;
  lastRefillDefault = Date.now();
  tokensChat = 0;
  lastRefillChat = Date.now();
  concurrentStreams = 0;
  streams = new Set<string>();
```

**Hydration:** first request loads state from storage under a mutex (`ensureStateLoaded`).

**Refill:** continuous token bucket — capacity = limit/min, refill rate = `capacity / 60000` tokens/ms.

### Non-SSE: `consume`

One token deducted per request. State persisted; returns `{ allow, headers }`.

### SSE: `open` / `close`

Two gates, checked in order:

1. **Concurrency cap**: if `streams.size >= concurrencyCap`, deny immediately (no token spent, stream not added).
2. **Chat token bucket**: same refill logic as default; on success, deduct 1 token and add `sid` to `streams`.

When the stream ends (or backend fails before streaming), `rlClose` removes `sid` from the set. The Worker schedules close via `ctx.waitUntil`:

- Backend fetch error before stream (502 path)
- Origin response not OK or no body
- Monitor loop on teed stream completes, times out (30 min), or errors — always calls `rlClose` in `finally`

`rlClose` is best-effort: errors are logged at debug level and do not fail the client response.

## Rate limit headers

The DO builds standard headers:

```108:116:gateway/src/rate_limiter.ts
  private headers(limit: number, remaining: number): Record<string, string> {
    const nowSec = Math.floor(Date.now() / 1000);
    const secs = new Date().getUTCSeconds();
    const reset = nowSec + (60 - secs);
    return {
      'X-RateLimit-Limit': String(limit),
      'X-RateLimit-Remaining': String(Math.max(0, Math.floor(remaining))),
      'X-RateLimit-Reset': String(reset),
    };
  }
```

**When headers appear on the client response:**

| Outcome | `X-RateLimit-*` on response? |
|---------|-------------------------------|
| **429 in enforce mode** | **No** — only CORS + `Content-Type: text/plain` via `errorCorsHeaders()` |
| Successful backend proxy | **Yes** — merged onto origin headers |
| SSE stream established | **Yes** — merged onto SSE response headers |
| Backend 502 (after RL check) | **Yes** — merged onto error response |
| DO timeout/error (fail open) | **No** — `rlHeaders` stays undefined |
| Cache HIT (after consume ran) | **No** — early return omits `rlHeaders` even though a token was consumed |

Documented in `docs/backpressure-guards.md`: no `Retry-After` is ever sent from the gateway.

## Fallback / fail-open behavior

Two distinct fallback paths:

### 1. DO unreachable or slow (>2s)

`rlConsume` / `rlOpen` use a 2-second `AbortController` timeout. On timeout or any thrown error:

- Log a warning with truncated hash and SSE context
- **Allow the request through** with no rate limit headers
- No 429

This is explicit fail-open at the edge (documented in `gateway/AGENTS.md`).

### 2. `observe` mode

DO runs normally; if over limit, `limited = true` but the request still proceeds. Analytics records the `limited` flag:

## End-to-end flow (ASCII)

```
Client → api.keplr.sh (Worker)
           │
           ├─ Public routes → passthrough (no RL)
           ├─ Bearer JWT → validate → backend (no RL)
           └─ X-API-Key
                 │
                 ├─ RL_MODE=off → skip
                 │
                 └─ RL_MODE=observe|enforce
                       │
                       ├─ /v1/chat/stream → rlOpen(chatPerMin + concurrency)
                       └─ everything else  → rlConsume(defaultPerMin)
                             │
                             ▼
                       RateLimiter DO (per sha256(apiKey))
                             │
                       ┌─────┴─────┐
                       │ timeout?  │──► fail open (no headers, no 429)
                       └─────┬─────┘
                             │
                       allow=false + enforce? ──► 429 (CORS only)
                             │
                       allow=false + observe? ──► continue (limited=1 in analytics)
                             │
                       allow=true ──► cache? ──► backend ──► attach X-RateLimit-* headers
                             │
                       SSE: tee stream, waitUntil(rlClose on end/timeout/error)
```

## Related but separate: backend auth rate limiting

The repo map also points at `crates/kepler-server/src/middleware/auth_rate_limit.rs`. That is a **different layer**: in-memory per-IP limits on auth bootstrap routes (`/v1/auth/register-client`, GitHub OAuth, service-token), applied inside the Rust API server after the gateway proxies the request. It is not part of the gateway DO rate limiter.
