Tracing the gateway rate limiter from route handling through limit headers and fallback behavior.
## Gateway rate limiter: end to end

The Cloudflare Worker gateway rate limiter is a **per-API-key** system built on a **Durable Object (DO)** token bucket. It runs only on the **`X-API-Key` path** after auth and scope checks. JWT Bearer requests, public passthrough routes, and `OPTIONS` never hit it.

There is a **separate** origin-side limiter in `auth_rate_limit.rs` for a few auth bootstrap endpoints (per-IP, in-memory). That is not part of the gateway DO flow below.

---

### 1. Request routing: who gets rate limited?

In `handleRequest` (`gateway/src/index.ts`), the handler walks branches in order:

1. `OPTIONS` → CORS only  
2. Public routes (OpenAPI, session exchange, health, JWKS, portal bootstrap, etc.) → proxy with **no** rate limit  
3. Auth required: `Authorization: Bearer` **or** `X-API-Key`  
4. **JWT path** (lines 772–835): validate JWT + scopes → proxy to backend → **no DO rate limit**  
5. **API-key path** (lines 838+): validate key + scopes → **rate limit** → cache → proxy

```843:901:gateway/src/index.ts
    const { cacheEnabled, rlMode, sseCap, defaultPerMin, chatPerMin } = limitsFromEnv(env);
    // ... scope enforcement ...
    const apiKeyHash = await sha256hex(apiKey);
    const sse = isSSE(url, request);
    // ...
    if (rlMode !== 'off') {
      try {
        if (sse) {
          sid = newSid();
          const r = await rlOpen(env, apiKeyHash, chatPerMin, sseCap, sid);
          // ...
        } else {
          const r = await rlConsume(env, apiKeyHash, defaultPerMin);
          // ...
        }
      } catch (e) {
        // Rate limiter timeout or error - log but allow request to proceed (fail open)
        console.warn('Rate limiter error, allowing request', { ... });
      }
    }
```

**Client identity for limits:** `sha256(apiKey)` → `env.RATE_LIMITER.idFromName(apiKeyHash)` — one DO instance per key hash.

**SSE detection:** only `/v1/chat/stream` (`isSSE`).

### 2. Configuration (`limitsFromEnv` + `wrangler.toml`)

| Env var | Default (prod `wrangler.toml`) | Role |
|---------|-------------------------------|------|
| `RL_MODE` | `observe` | `off` \| `observe` \| `enforce` |
| `RL_DEFAULT_LIMIT` | `1000` | Regular requests/min (token bucket capacity) |
| `RL_CHAT_LIMIT` | `100` | Chat stream starts/min |
| `SSE_CONCURRENCY_LIMIT` | `5` | Max concurrent open SSE streams per key |

```543:550:gateway/src/index.ts
function limitsFromEnv(env: Env) {
  return {
    cacheEnabled: (env.CACHE_ENABLED || 'false').toLowerCase() === 'true',
    rlMode: (env.RL_MODE || 'off').toLowerCase() as 'off' | 'observe' | 'enforce',
    sseCap: Number(env.SSE_CONCURRENCY_LIMIT || '5'),
    defaultPerMin: Number(env.RL_DEFAULT_LIMIT || '1000'),
    chatPerMin: Number(env.RL_CHAT_LIMIT || '100'),
  };
}
```

The DO is bound as `RATE_LIMITER` in `gateway/wrangler.toml` (SQLite-backed migration `v1-rate-limiter`).

### 3. Worker → Durable Object RPC

Three thin wrappers POST JSON to the DO stub with a **2s abort timeout**:

| Function | Payload | When |
|----------|---------|------|
| `rlConsume` | `{ type: 'consume', limits: { defaultPerMin } }` | Non-SSE API-key requests |
| `rlOpen` | `{ type: 'open', sid, limits: { chatPerMin, concurrency } }` | SSE stream start |
| `rlClose` | `{ type: 'close', sid }` | Stream end / failure (best-effort via `ctx.waitUntil`) |

```555:575:gateway/src/index.ts
async function rlConsume(env: Env, apiKeyHash: string, defaultPerMin: number) {

  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), 2000);
  // ... stub.fetch POST → { allow, headers }
}
```

### 4. Durable Object algorithm (`RateLimiter`)

`RateLimiter.fetch` accepts POST only, hydrates persisted state once (mutex via `hydrationPromise`), then dispatches on `payload.type`.

**State (persisted in DO storage):**

- `tokensDefault` / `lastRefillDefault` — general API bucket  
- `tokensChat` / `lastRefillChat` — chat-start bucket  
- `streams` — `Set<string>` of active SSE session IDs (concurrency)

**Refill:** continuous token bucket — capacity = limit/min, refill rate = `capacity / 60000` tokens/ms.

```99:106:gateway/src/rate_limiter.ts
  private refill(now: number, limitPerMin: number, lastRefill: number, tokens: number): { tokens: number; lastRefill: number } {
    const capacity = limitPerMin;
    const elapsed = Math.max(0, now - lastRefill);
    const rate = capacity / 60000;
    const newTokens = Math.min(capacity, tokens + elapsed * rate);
    return { tokens: newTokens, lastRefill: now };
  }
```

**`consume`:** refill default bucket → if ≥ 1 token, decrement and `allow: true` → persist → return `{ allow, headers }`.

**`open` (SSE):**

1. If `streams.size >= concurrencyCap` → `allow: false` immediately (no token spent)  
2. Else refill chat bucket → if ≥ 1 token, decrement, add `sid` to `streams`, `allow: true`  
3. Persist chat tokens + streams array

**`close`:** remove `sid` from `streams`, persist (idempotent).

### 5. Rate-limit headers

Computed in `RateLimiter.headers`:

```108:116:gateway/src/rate_limiter.ts
  private headers(limit: number, remaining: number): Record<string, string> {
    const nowSec = Math.floor(Date.now() / 1000);
    const secs = new Date().getUTCSeconds();
    const reset = nowSec + (60 - secs);
    return {
      'X-RateLimit-Limit': String(limit),
      'X-RateLimit-Remaining': String(Math.max(0, Math.floor(remaining))),
      'X-RateLimit-Reset': String(reset),
    };
  }
```

- **`X-RateLimit-Limit`:** bucket capacity (default or chat limit depending on path)  
- **`X-RateLimit-Remaining`:** floored token count after this operation  
- **`X-RateLimit-Reset`:** Unix timestamp for end of current UTC minute  
- **`Retry-After`:** never sent (documented in `docs/backpressure-guards.md`)

Headers are merged onto **successful proxy responses**, SSE responses, and **502 Bad Gateway** when the backend fetch fails:

```978:980:gateway/src/index.ts
    const headers = new Headers(originRes.headers);
    if (rlHeaders) for (const [k, v] of Object.entries(rlHeaders)) headers.set(k, v);
```

**429 responses do not include rate-limit headers** — they use plain CORS + `text/plain`:

```884:889:gateway/src/index.ts
          if (rlMode === 'enforce' && !r.allow) return new Response('Too Many Requests', { status: 429, headers: errorCorsHeaders() });
```

**Cache HIT path** also omits `X-RateLimit-*` (rate was consumed, but headers are not attached on the early return at ~line 934).

### 6. Mode behavior: observe vs enforce vs off

| Mode | DO called? | Over limit | Headers on success |
|------|------------|------------|-------------------|
| `off` | No | N/A | None from limiter |
| `observe` | Yes | Request **still proceeds**; `limited=true` logged to Analytics Engine | Yes |
| `enforce` | Yes | **429** immediately, CORS only | N/A (blocked) |

In `observe`, the DO decision is recorded via Analytics (`doubles: [status, limited ? 1 : 0]`) but the client is never rejected.

### 7. Fallback / fail-open behavior

If the DO call **times out (2s)** or throws for any other reason:

```891:900:gateway/src/index.ts
      } catch (e) {
        console.warn('Rate limiter error, allowing request', { ... });
        // Continue without rate limiting headers
      }
```

- Request **proceeds** (fail open)  
- No `X-RateLimit-*` headers  
- No 429 from the limiter  

`rlClose` failures are similarly non-fatal — logged at `debug`, never fail the client response.

### 8. SSE concurrency lifecycle

```mermaid
sequenceDiagram
  participant Client
  participant Worker as gateway/index.ts
  participant DO as RateLimiter DO
  participant Backend

  Client->>Worker: POST /v1/chat/stream (X-API-Key)
  Worker->>DO: open(sid, chatPerMin, concurrency)
  alt over cap or no tokens
    DO-->>Worker: allow=false
    Worker-->>Client: 429 (enforce only)
  else allowed
    DO-->>Worker: allow=true + headers
    Worker->>Backend: proxy SSE
    Backend-->>Worker: stream body
    Worker-->>Client: stream + X-RateLimit-*
    Note over Worker: tee body; waitUntil monitors stream
    Worker->>DO: close(sid) on done/timeout/error
  end
```

- **`open`** reserves a concurrency slot + consumes a chat token  
- **`close`** runs on: stream completion, 30-minute monitor timeout, backend error before body established, or failed backend fetch  
- Concurrency slot is released in `finally` of the background monitor (lines 1005–1033)

### 9. Full flow (API-key request)

```
Client → Worker fetch()
  → handleRequest()
    → [public?] skip
    → [Bearer JWT?] skip rate limit
    → validate X-API-Key + scopes
    → sha256(apiKey) → DO id
    → [RL_MODE=off] skip
    → [SSE] rlOpen / [else] rlConsume  (2s timeout)
         → [error/timeout] fail open, no headers
         → [enforce && !allow] 429 (CORS only)
         → [observe && !allow] continue, limited=1 in analytics
    → [cacheable GET] maybe cache HIT (no RL headers on hit)
    → proxy to BACKEND_URL
    → attach X-RateLimit-* to response (unless cache hit / 429)
    → [SSE] tee stream + waitUntil(rlClose)
```
