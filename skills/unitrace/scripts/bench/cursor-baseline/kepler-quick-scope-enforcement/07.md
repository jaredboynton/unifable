Tracing gateway scope lookup and enforcement through the codebase.
## Overview

Gateway scope enforcement is a two-step pipeline at the Cloudflare Worker edge:

1. **Lookup** — `getRequiredScopes(pathname, method)` maps the request path (and sometimes HTTP method) to required scope(s), using generated helpers from `policy/scope-matrix.json`.
2. **Enforce** — After auth succeeds, `hasScope(granted, required)` checks whether the caller’s granted scopes satisfy every required scope. Failure returns **403**; there is no silent downgrade.

The canonical policy lives in `policy/scope-matrix.json`. Codegen (`policy/generate.mjs`, run via `just generate-scope-matrix`) emits `gateway/src/generated/scope-matrix.ts` (and `.js`). The gateway must not hand-edit those files.

---

## 1. Where route→scope mappings come from

`policy/scope-matrix.json` defines:

- **`routes`** — path prefixes (and a few method-specific admin read exceptions)
- **`public_backend_passthrough`** — backend routes that skip gateway auth
- **`public_gateway_routes`** — gateway-handled public routes (OpenAPI, `/health`, session exchange)
- **`aliases`** — accepted alternate scope names (e.g. `kepler:communications:read` satisfies `kepler:communications:content:read`)

Example route entries:

        ...
        {
            "path": "/v1/salesforce/",

## 2. How lookup works (`getRequiredScopes`)

The gateway imports generated helpers in `gateway/src/index.ts`:

```51:59:gateway/src/index.ts
import {
  getRequiredScopes,
  hasScope,
  isPublicBackendPassthroughPath,
  isPublicGatewayBackendPassthroughPath,
  isPublicGatewayOpenApiPath,
  isPublicGatewaySessionExchangePath,
} from './generated/scope-matrix';
```

`getRequiredScopes` logic:

1. **Admin read exceptions** — three exact paths require `kepler:admin:read` on `GET`, `kepler:admin:write` otherwise.
2. **Longest-prefix wins** — `ROUTE_SCOPES` entries are sorted by path length descending; first exact match or prefix match wins (so `/v1/kg/admin/` beats `/v1/kg/`).
3. **No match → `null`** — no gateway scope check for that path (backend may still enforce per-route middleware).

```217:225:gateway/src/generated/scope-matrix.ts
export function getRequiredScopes(pathname: string, method: string): string | string[] | null {
  if (pathname === '/v1/admin/providers/health' || pathname === '/v1/admin/providers/config' || pathname === '/v1/admin/diagnostics/scope-mismatch')
    return method === 'GET' ? ADMIN_READ_SCOPE : ADMIN_WRITE_SCOPE;
  const sorted = Object.entries(ROUTE_SCOPES).sort((a, b) => b[0].length - a[0].length);
  for (const [prefix, scope] of sorted) {
    if (pathname === prefix || pathname.startsWith(prefix)) return scope;
  }
  return null;
}
```

Public-route helpers (`isPublicBackendPassthroughPath`, etc.) use `matchesPathPolicy` with `exact`, `prefix`, and `prefix_suffix` rules — those paths never reach scope enforcement.

## 3. Request flow: when scope is checked

Entry point: `handleRequest` in `gateway/src/index.ts` (called from the Worker `fetch` handler).

Order:

| Stage | What happens | Scope check? |
|-------|----------------|--------------|
| `OPTIONS` | CORS preflight | No |
| Public OpenAPI | Serve bundled spec locally | No |
| `/v1/auth/session` | Okta introspection + backend proxy | No |
| `/health` (gateway public) | Backend passthrough | No |
| Auth bootstrap / portal / JWKS | `isPublicBackendPassthroughPath` | No |
| Authenticated traffic | Bearer JWT **or** `X-API-Key` | **Yes** |

Public paths return before credentials are required. Scope enforcement runs only on authenticated branches.

## 5. Scope matching (`hasScope`)

Shared matching logic in generated code:

```227:239:gateway/src/generated/scope-matrix.ts
export function hasScope(granted: string, required: string): boolean {
  if (!required) return false;
  const grantedScopes = granted.split(/\s+/);
  const matches = (req: string): boolean =>
    grantedScopes.some((s) => {
      if (s === req) return true;
      if (s.endsWith(':*') && req.startsWith(s.slice(0, -1))) return true;
      return false;
    });
  if (matches(required)) return true;
  const aliases = SCOPE_ALIASES[required];
  if (aliases) return aliases.some((alt) => matches(alt));
  return false;
}
```

Supports:

- Exact scope tokens in a whitespace-separated grant string
- Wildcards like `kepler:admin:*`
- Aliases (currently `kepler:communications:content:read` accepts `kepler:communications:read`)

The backend uses the same policy and equivalent `has_scope` (via `crates/kepler-server/src/middleware/scopes.rs` → generated `scope_matrix.rs`), but attaches scopes at **route registration** time (`require_auth(Some(SCOPE_...))` in `main.rs`) rather than dynamic path lookup. The gateway does dynamic lookup so it can reject bad requests before proxying.

## 6. Important behaviors and edge cases

1. **Fail-closed on mapped routes** — If a route maps to a scope and the caller lacks it, the gateway returns 403; it does not proxy and hope the backend catches it.
2. **Unmapped routes pass gateway scope checks** — `getRequiredScopes` returning `null` skips enforcement at the edge; backend middleware may still require scopes on those handlers.
3. **Public routes skip both auth and scope** — Including auth bootstrap (`/v1/auth/validate`, `/v1/auth/service-token`, portal routes, JWKS, etc.).
4. **Dual layer** — Gateway enforces at the edge; backend re-enforces on mounted routes. Same vocabulary and aliases from one policy file.
5. **Regeneration required** — Route/scope changes go in `policy/scope-matrix.json`, then `just generate-scope-matrix`; CI checks generated artifacts stay in sync.
